# P4 — Infrastructure Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Provision all Azure infrastructure via Terraform and add deploy + tf-plan GitHub Actions workflows so the complete app (P2 BFF + P3 SPA) can be deployed to staging and production with one push.

**Architecture:** Single Azure Container App (min=0, max=1) behind Container Apps ingress; user-assigned Managed Identity for Key Vault + ACR access; azurerm backend in Storage Account; OIDC federated credentials for CI/deploy, never client secrets.

**Tech Stack:** Terraform ≥1.7, azurerm provider ~>3.100, azuread provider ~>2.50, GitHub Actions with az CLI + OIDC login.

**Reference:** `docs/superpowers/specs/2026-04-24-spectra-design.md` §8, §9, §10, §11, §12, `CLAUDE.md` §7.

**Deliverable at P4 end:**
- `terraform plan` and `terraform apply` succeed on a fresh subscription against the `dev` workspace.
- Container App boots from an ACR image, fetches secrets from Key Vault via Managed Identity, and `/health` + `/ready` smoke tests pass against the public ingress URL.
- `deploy.yml` deploys to staging on merge to `main` and to prod on a `v*` release tag (with reviewer gate).
- `tf-plan.yml` posts a Terraform plan as a PR comment for any PR touching `infra/`.
- `ci.yml` pushes the built image to ACR with `:sha-<sha>` tag after a successful Trivy scan.
- Runbooks for tenant setup, rollback, and secret rotation are committed.

## Phase A — Terraform skeleton & bootstrap

### Task A1: Create infra directory and bootstrap script for Storage Account state

**Files:**
- Create: `infra/.gitkeep`
- Create: `scripts/bootstrap-tfstate.sh`

The azurerm backend must point at a Storage Account that exists *before* `terraform init` runs. We create that Storage Account with a one-shot bash script per environment (dev/staging/prod). After this script runs once, all further state is managed by Terraform itself.

- [ ] **Step 1: Create the directory**

```bash
mkdir -p infra/modules
touch infra/.gitkeep
```

- [ ] **Step 2: Write the bootstrap script**

Create `scripts/bootstrap-tfstate.sh`:
```bash
#!/usr/bin/env bash
# Bootstraps the Azure Storage Account that holds Terraform remote state.
# Run once per Azure subscription (or per environment if you keep state separate).
# Required env: AZURE_SUBSCRIPTION_ID, ENVIRONMENT (dev|staging|prod), LOCATION.
set -euo pipefail

: "${AZURE_SUBSCRIPTION_ID:?AZURE_SUBSCRIPTION_ID required}"
: "${ENVIRONMENT:?ENVIRONMENT required (dev|staging|prod)}"
: "${LOCATION:?LOCATION required (e.g. canadacentral)}"

RG_NAME="rg-spectra-tfstate-${ENVIRONMENT}"
SA_NAME="stspectratf$(echo -n "${ENVIRONMENT}${AZURE_SUBSCRIPTION_ID}" | shasum -a 256 | cut -c1-8)"
CONTAINER_NAME="tfstate"

echo "Subscription : ${AZURE_SUBSCRIPTION_ID}"
echo "Resource grp : ${RG_NAME}"
echo "Storage acct : ${SA_NAME}"
echo "Container    : ${CONTAINER_NAME}"

az account set --subscription "${AZURE_SUBSCRIPTION_ID}"

az group create \
  --name "${RG_NAME}" \
  --location "${LOCATION}" \
  --tags environment="${ENVIRONMENT}" project=spectra managed_by=bootstrap

az storage account create \
  --name "${SA_NAME}" \
  --resource-group "${RG_NAME}" \
  --location "${LOCATION}" \
  --sku Standard_LRS \
  --kind StorageV2 \
  --access-tier Hot \
  --https-only true \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false \
  --tags environment="${ENVIRONMENT}" project=spectra managed_by=bootstrap

az storage container create \
  --name "${CONTAINER_NAME}" \
  --account-name "${SA_NAME}" \
  --auth-mode login

cat <<EOF

Bootstrap complete.

Add the following to infra/backend.tf (or pass as -backend-config):
  resource_group_name  = "${RG_NAME}"
  storage_account_name = "${SA_NAME}"
  container_name       = "${CONTAINER_NAME}"
  key                  = "spectra.${ENVIRONMENT}.tfstate"

EOF
```

- [ ] **Step 3: Mark executable**

```bash
chmod +x scripts/bootstrap-tfstate.sh
```

- [ ] **Step 4: Commit**

```bash

git commit -m "chore(infra): scaffold infra dir and tfstate bootstrap script"
```

**Files:**
- Create: `infra/modules/keyvault/main.tf`
- Create: `infra/modules/keyvault/variables.tf`

```hcl
resource "azurerm_key_vault" "main" {
  name                          = var.name
  resource_group_name           = var.resource_group_name
  location                      = var.location
  tenant_id                     = var.tenant_id
  sku_name                      = "standard"
  enable_rbac_authorization     = true
  purge_protection_enabled      = true

  public_network_access_enabled = true

  network_acls {
    default_action = "Allow"
    bypass         = "AzureServices"
  }

# Deployer (Terraform CI principal or local operator) needs to write the AAD client secret.
resource "azurerm_role_assignment" "deployer_secrets_officer" {
  scope                = azurerm_key_vault.main.id
  role_definition_name = "Key Vault Secrets Officer"
  principal_id         = var.deployer_object_id
  description          = "Allows Terraform deployer to manage secret values."

resource "azurerm_key_vault_secret" "aad_client_secret" {

  value        = "placeholder-overwritten-by-aad-app-module"
  key_vault_id = azurerm_key_vault.main.id

  content_type = "AAD client secret"

  depends_on = [azurerm_role_assignment.deployer_secrets_officer]

  lifecycle {
    ignore_changes = [value, version]
  }
}

resource "azurerm_key_vault_secret" "cookie_hmac_key" {
  name         = "cookie-hmac-key"
  value        = "placeholder-rotate-with-rotation-runbook"

  depends_on = [azurerm_role_assignment.deployer_secrets_officer]

    ignore_changes = [value, version]
  }
}

resource "azurerm_key_vault_secret" "session_encryption_key" {
  name         = "session-encryption-key"
```

## Phase I — Root composition

### Task I1: Resource group + module composition + outputs

**Files:**
- Create: `infra/main.tf`
- Create: `infra/outputs.tf`

The root composition wires everything together. There is a circular dependency between `keyvault` (creates the placeholder secret) and `aad-app` (writes the real value), resolved by giving each module a dedicated `azurerm_key_vault_secret` resource with a different name in its lifecycle block — the `aad-app` module writes the canonical value, the `keyvault` module owns rotation-safe placeholders for the cookie + session keys.

To avoid the placeholder vs. real-value collision on `aad-client-secret`, the `keyvault` module's placeholder for that one secret is dropped from the resource set when `aad-app` runs in the same plan. We resolve this by having the `keyvault` module *not* create the `aad-client-secret` placeholder; instead, only `aad-app` creates that secret. The remaining two secrets (`cookie-hmac-key`, `session-encryption-key`) keep their placeholders in `keyvault` because no module ever writes them — operators rotate them manually per the runbook.

Create `.github/workflows/deploy.yml`:
```yaml

  push:
    branches: [main]
    tags: ['v*']

permissions:
  id-token: write   # OIDC for az login
  contents: read

concurrency:
  group: deploy-${{ github.ref }}
  cancel-in-progress: false

jobs:
  resolve-env:
    name: Resolve target environment
    runs-on: ubuntu-latest
    outputs:
      environment: ${{ steps.pick.outputs.environment }}
      tfvars_file: ${{ steps.pick.outputs.tfvars_file }}
    steps:
      - id: pick
        run: |
          if [[ "${GITHUB_REF}" == refs/tags/v* ]]; then
            echo "environment=production" >> "$GITHUB_OUTPUT"
            echo "tfvars_file=env/prod.tfvars" >> "$GITHUB_OUTPUT"
          else
            echo "environment=staging" >> "$GITHUB_OUTPUT"
            echo "tfvars_file=env/staging.tfvars" >> "$GITHUB_OUTPUT"
          fi

  deploy:
    needs: resolve-env
    name: Deploy to ${{ needs.resolve-env.outputs.environment }}
    runs-on: ubuntu-latest
    environment: ${{ needs.resolve-env.outputs.environment }}
    timeout-minutes: 30
    env:
      ARM_USE_OIDC: "true"
      ARM_CLIENT_ID: ${{ secrets.AZURE_CLIENT_ID }}
      ARM_TENANT_ID: ${{ secrets.AZURE_TENANT_ID }}
      ARM_SUBSCRIPTION_ID: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      TF_VAR_subscription_id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
      TF_VAR_tenant_id: ${{ secrets.AZURE_TENANT_ID }}
      TF_VAR_github_org: ${{ github.repository_owner }}
      TF_VAR_image_tag: sha-${{ github.sha }}
    steps:
      - uses: actions/checkout@v4

      - name: Azure login (OIDC)
        uses: azure/login@v2
        with:
          client-id: ${{ secrets.AZURE_CLIENT_ID }}
          tenant-id: ${{ secrets.AZURE_TENANT_ID }}
          subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

        with:
          terraform_version: 1.7.5

      - name: Terraform init
        working-directory: infra
        run: |
          terraform init \
            -backend-config="resource_group_name=${{ secrets.TFSTATE_RG }}" \
            -backend-config="storage_account_name=${{ secrets.TFSTATE_SA }}" \
            -backend-config="container_name=tfstate" \
            -backend-config="key=spectra.${{ needs.resolve-env.outputs.environment == 'production' && 'prod' || 'staging' }}.tfstate"

      - name: Terraform plan
        working-directory: infra
        run: |
          terraform plan \
            -var-file=${{ needs.resolve-env.outputs.tfvars_file }} \
            -out=tfplan

      - name: Terraform apply
        working-directory: infra
        run: terraform apply -auto-approve tfplan

      - name: Capture outputs
        id: tf
        working-directory: infra
        run: |
          {
            echo "ACR_LOGIN_SERVER=$(terraform output -raw acr_login_server)"
            echo "ACR_NAME=$(terraform output -raw acr_name)"
            echo "CA_NAME=$(terraform output -raw container_app_name)"
            echo "RG_NAME=$(terraform output -raw resource_group_name)"
            echo "FQDN=$(terraform output -raw container_app_fqdn)"
          } >> "$GITHUB_OUTPUT"

      - name: Update Container App image
        run: |
          az containerapp update \
            --name "${{ steps.tf.outputs.CA_NAME }}" \
            --resource-group "${{ steps.tf.outputs.RG_NAME }}" \
            --image "${{ steps.tf.outputs.ACR_LOGIN_SERVER }}/spectra:sha-${{ github.sha }}"

      - name: Wait for revision to become ready
        run: |
          for i in {1..30}; do
            STATE=$(az containerapp revision list \
              --name "${{ steps.tf.outputs.CA_NAME }}" \
              --resource-group "${{ steps.tf.outputs.RG_NAME }}" \
              --query "[?properties.active].properties.healthState | [0]" -o tsv)
            echo "Revision health: ${STATE}"
            if [[ "${STATE}" == "Healthy" ]]; then
              exit 0
            fi
            sleep 10
          done
          echo "Revision did not reach Healthy within 5 minutes" >&2
          exit 1

      - name: Smoke test /health
        run: |
          curl --fail --silent --show-error --retry 5 --retry-delay 10 \
            "https://${{ steps.tf.outputs.FQDN }}/health"

      - name: Smoke test /ready
        run: |
          curl --fail --silent --show-error --retry 5 --retry-delay 10 \
            "https://${{ steps.tf.outputs.FQDN }}/ready"
```

````markdown
# Rollback runbook

Use when a deployed revision is misbehaving (5xx spike, wrong config, broken UI). The Container App keeps the last 100 revisions; rollback is a traffic swap — no rebuild required.

## When to roll back

Roll back if any of:
- `requests/failed` rate alert fires within 5 minutes of deploy.
- `/health` or `/ready` returns non-2xx for > 2 minutes after revision becomes Active.
- Auth failure spike alert fires on the new revision.
- Manual report of broken behavior with high blast radius.

## Procedure (≤ 2 minutes)

```bash
RG=rg-spectra-prod        # or staging
CA=ca-spectra-prod         # match your env

# 1. Find the previous healthy revision
az containerapp revision list \
  --name "$CA" --resource-group "$RG" \
  --query "sort_by([?properties.healthState=='Healthy'],&properties.createdTime)[*].{name:name,created:properties.createdTime,active:properties.active}" \
  -o table

PREV=<name from list — the one before the current active>

# 2. Pin traffic to the previous revision
az containerapp revision set-mode \
  --name "$CA" --resource-group "$RG" \
  --mode Single

az containerapp ingress traffic set \
  --name "$CA" --resource-group "$RG" \
  --revision-weight "${PREV}=100"

# 3. Verify
FQDN=$(az containerapp show --name "$CA" --resource-group "$RG" --query properties.configuration.ingress.fqdn -o tsv)
curl --fail "https://${FQDN}/health"
curl --fail "https://${FQDN}/ready"
```

Open an incident, link the deploy that introduced the regression, and post in #spectra-incidents.

## Re-deploy after a fix

After fixing the regression on `main`:
1. Let `ci.yml` push the new image as `sha-<sha>`.
2. Either re-run `deploy.yml` (push to main) or manually:
   ```bash
   az containerapp update \
     --name "$CA" --resource-group "$RG" \
     --image "<acr>/spectra:sha-<new-sha>"
   ```
3. Smoke test `/health` + `/ready` before removing traffic pin.

## Rolling back Terraform changes

If the regression came from infra (not the image):
```bash
git revert <bad-commit>
git push origin main
# deploy.yml will run terraform apply on the reverted state
```

For state corruption, restore from the Storage Account's blob versioning:
```bash
az storage blob list \
  --account-name <TFSTATE_SA> \
  --container-name tfstate \
  --query "[?name=='spectra.prod.tfstate'].{name:name,versionId:versionId,modified:properties.lastModified}" \
  -o table

az storage blob copy start \
  --account-name <TFSTATE_SA> \
  --destination-container tfstate \
  --destination-blob spectra.prod.tfstate \
  --source-uri "https://<TFSTATE_SA>.blob.core.windows.net/tfstate/spectra.prod.tfstate?versionid=<good-version>"
```
````

````markdown
# Secret rotation runbook

Spectra has three secrets in Key Vault. Each rotates on a different cadence and has a different impact on running sessions.

| Secret | Cadence | Session impact |
|---|---|---|
| `aad-client-secret` | 12 months (or on suspected leak) | None — server-side only |
| `cookie-hmac-key` | 6 months | All sessions invalidated; users sign in again |
| `session-encryption-key` | 6 months | All sessions invalidated; users sign in again |

The Container App always reads the **latest** version of each secret because the env var references use the versionless URI. A revision restart picks up the new value.

## Operator prerequisites

- Member of the deployer SP or holder of `Key Vault Secrets Officer` on the target vault.
- `az` CLI logged in.

## 1. Rotate `aad-client-secret`

The Terraform-managed `azuread_application_password` resource owns this. Rotation is by Terraform re-creation:

```bash
cd infra
terraform init -backend-config=...
terraform taint module.aad_app.azuread_application_password.main
terraform apply -var-file=env/prod.tfvars
```

The new password is written to KV in the same plan. Restart the Container App to pick it up:

```bash
az containerapp revision restart \
  --name ca-spectra-prod \
  --resource-group rg-spectra-prod \
  --revision $(az containerapp revision list \
    --name ca-spectra-prod --resource-group rg-spectra-prod \
    --query "[?properties.active]|[0].name" -o tsv)
```

## 2. Rotate `cookie-hmac-key` (sessions invalidated)

Generate 64 random bytes, base64-url-encode, and write to KV:

```bash
NEW_KEY=$(openssl rand -base64 64 | tr -d '=' | tr '/+' '_-' | tr -d '\n')
az keyvault secret set \
  --vault-name <kv-name> \
  --name cookie-hmac-key \
  --value "${NEW_KEY}"
```

Restart the Container App as in step 1. All existing session cookies fail signature verification → users redirected to login.

Announce the rotation in #spectra-announce ahead of time so users expect to re-sign-in.

## 3. Rotate `session-encryption-key` (sessions invalidated)

Generate 32 random bytes for AES-256:

```bash
NEW_KEY=$(openssl rand -base64 32 | tr -d '=' | tr '/+' '_-' | tr -d '\n')
az keyvault secret set \
  --vault-name <kv-name> \
  --name session-encryption-key \
  --value "${NEW_KEY}"
```

Restart the Container App. Existing session JSON in the SPE system container becomes undecryptable; sessions are re-created on next sign-in. Optionally, run the SPE cleanup script to delete stale `/sessions/*.json`:

```bash
# (Requires admin token; uses the BFF cleanup endpoint scaffolded in P2)
curl -X POST "https://<fqdn>/api/admin/sessions/purge" \
  -H "Authorization: Bearer <admin-id-token>"
```

## 4. Verify

After every rotation:
- `curl https://<fqdn>/ready` returns `200`.
- App Insights shows no `KeyVaultAccessFailed` events in the last 5 minutes.
- A test sign-in succeeds end-to-end.

## 5. Emergency rotation (suspected leak)

1. Rotate all three secrets in the order: `aad-client-secret` → `cookie-hmac-key` → `session-encryption-key`.
2. Disable the previous AAD client secret in Entra ID (App registration → Certificates & secrets → Delete the old credential).
3. Open an incident; preserve App Insights logs for the suspected window.
4. Notify users that they need to sign in again.
````
